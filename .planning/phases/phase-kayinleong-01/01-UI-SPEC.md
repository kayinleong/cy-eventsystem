# Phase 1 — UI Design Contract

> Visual and interaction contract for the UI POC phase of cy-eventsystem. Every route in the sitemap must render against this contract using mock data only. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

| Property | Value |

| Tool | shadcn/ui v4.8.0 |

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-text gaps, badge inner padding, status-dot offset |
| sm | 8px | Compact stack spacing, form-field inner gaps |
| md | 16px | Default element spacing, card padding, table cell horizontal padding |
| lg | 24px | Section padding, between form groups, card-to-card gaps |
| xl | 32px | Layout gaps, page top padding |
| 2xl | 48px | Major section breaks (e.g. dashboard hero → cards) |
| 3xl | 64px | Page-level spacing (top of dashboard, empty-state vertical padding) |

### Density

| Surface | Row / Item Height | Notes |
|---------|-------------------|-------|
| Table row (default) | 40px | Comfortable density, ~18 rows per viewport on 13" laptop |
| Table row (list pages — `/inventory`, `/events`, `/users`, `/reports/*`) | 32–40px | Tight enough to scan; never below 32px |
| Form field input | 36px (default `h-9`) | shadcn default |
| Button (default) | 36px (`h-9`) | shadcn default |

### Exceptions

- **44px minimum touch target** for: scanner action buttons (`/scan`, `/events/[id]/checkout`, `/events/[id]/checkin`), quantity steppers (qty +/- on check-out cart and inventory edit). WCAG 2.5.5 Level AAA target size compliance for one-handed mobile use.
- List row heights MAY drop to 32px when density toggle is added in a future phase; Phase 1 ships at 40px comfortable.

## Typography

Font families: **Geist Sans** (UI text), **Geist Mono** (SKUs, IDs, scanned codes).

| Role | Size | Tailwind | Weight | Line Height | Usage |
|------|------|----------|--------|-------------|-------|
| Body | 14px | `text-sm` | 400 (regular) | 1.5 | Default paragraph, table cells, form helper text, list rows |
| Label / Heading-S | 14px | `text-sm` | 600 (semibold) | 1.4 | Form labels, table column headers, card titles, nav items |
| Heading-M | 18px | `text-lg` | 600 (semibold) | 1.3 | Page titles, section headings, dialog titles, sheet titles |
| Display | 24px | `text-2xl` | 600 (semibold) | 1.2 | Dashboard hero counts, scanner result code, large summary numbers |

Only **two weights** ship in Phase 1: `400` regular and `600` semibold. No italic. Mono only for codes (SKU, item ID, scanned barcode, event ID).

## Color

Monochrome neutral base with semantic status colors as the only chromatic ink. All values come from the radix-nova / neutral CSS variables already in `app/globals.css`.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` | Page background, default surface |
| Secondary (30%) | `--card`, `--secondary`, `--muted` | Cards, sidebar, nav rails, table header strip |
| Accent (10%) | `--primary` (near-black light / near-white dark) | Reserved list below |
| Destructive | `--destructive` | Destructive actions and `missing` status only |
| Foreground | `--foreground` | Default text |
| Muted foreground | `--muted-foreground` | Helper text, placeholders, secondary metadata |
| Border | `--border` | All hairline rules, table dividers, input outlines |
| Ring | `--ring` | Focus ring (uses primary) |

### Accent reserved exclusively for

1. **The single primary CTA per surface** (e.g. `[Add item]` on `/inventory`, `[Create event]` on `/events`, `[Check out]` on `/events/[id]/checkout`).
2. **Active nav item** in the sidebar / top nav (current route).
3. **Focus ring** on keyboard-focused interactive elements.
4. **Current-page breadcrumb** segment.

Accent must NOT be used for: secondary buttons, hyperlinks in body text, icon defaults, badges, hover states, table row hover, or "you have unread X" indicators. Secondary actions use `variant="outline"` or `variant="ghost"`.

### Status Palette (Q4)

Style: **outline badge + colored dot** (Linear-style — calm, not screaming). The badge text inherits foreground; the dot carries the semantic color.

| Status | Dot color | Token / Tailwind | Applies to |
|--------|-----------|------------------|------------|
| available / planned / active | green | `bg-green-500` (light) / `bg-green-400` (dark) | inventory item status, event status |
| checked_out / in-progress | blue | `bg-blue-500` / `bg-blue-400` | inventory item status, event status, scan-cart item |
| damaged / low-stock / overdue | amber | `bg-amber-500` / `bg-amber-400` | inventory condition, stock thresholds, event return overdue |
| retired / completed / cancelled | muted gray | `bg-muted-foreground` | terminal states |
| missing | red (destructive) | `bg-destructive` | unreturned items past grace, audit gaps |

Only `missing` may use the destructive token in the status system. All other "bad" states use amber. Destructive button styling is reserved for irreversible actions (retire, cancel event, disable user, write off).

## Layout & Route Patterns

| Pattern | Used for | Rationale |
|---------|----------|-----------|
| Full-page route | `/inventory/new`, `/inventory/[id]/edit`, `/events/new`, `/users/new` (invite) | Long forms, multi-section, need URL state |
| `<Sheet/>` (right slide-over) | Short forms — quick edits (e.g. tag rename, threshold edit), event detail side panel | Stays in-context, no nav cost for small edits |
| `<Dialog/>` modal | Destructive confirmations only | Forces deliberate dismissal |
| `<ScannerModal/>` (single shared component) | `/scan`, `/events/[id]/checkout`, `/events/[id]/checkin` | One camera implementation, one decode pipeline, mode-aware (check-out / check-in / lookup) |
| `<DataTable/>` (shadcn data-table block + TanStack v8) | All list pages: `/inventory`, `/events`, `/users`, `/reports/*` | Consistent sorting, filtering, pagination, column visibility |
| Card grid | Dashboard (`/`) — KPI summary cards above recent activity table | Hero numbers must be glanceable |

### App shell

- Persistent left sidebar (collapsible) on `md+`, bottom nav on `<md` (mobile).
- Sidebar items: Dashboard / Inventory / Scan / Events / Reports / Users (admin) / Settings.
- Top bar: breadcrumbs (left), search (center, where applicable), user menu (right) containing theme toggle (light / dark / system) + sign out.
- All content max-width: `1400px` centered with `px-6` on `md+`, `px-4` on mobile.

## Dark Mode

- Manual toggle via `next-themes` in the user menu. Options: **Light**, **Dark**, **System** (default).
- Theme attribute: `class` on `<html>`.
- Both `:root` and `.dark` token sets confirmed present in `app/globals.css`.
- Every surface must be designed and verified in both modes during Phase 1 implementation. No dark-mode-only or light-mode-only screens.
- Status dot colors swap to the lighter variant in `.dark` (see Status Palette table).

## Voice & Copywriting

### Voice rules

- **Direct, operational, verb-noun.** "Add item" not "Add an item" or "Let's add an item".
- No conversational softeners ("Oops!", "Hmm…", "Just", "Please").
- No exclamation marks except for success toasts where the celebratory tone aids feedback (sparingly — max one per toast).
- Sentence case everywhere. **Never** title case in body, buttons, or labels. Page titles are sentence case (`Inventory`, not `INVENTORY`).
- Numbers: digits, not words (`3 items`, not `three items`).
- Dates: relative under 7 days (`2 hours ago`, `Yesterday`); absolute beyond (`May 12, 2026`).

### Primary CTA labels (per surface)

| Surface | Primary CTA |
|---------|-------------|
| `/inventory` | Add item |
| `/inventory/[id]` | Edit |
| `/events` | Create event |
| `/events/[id]` (planned) | Start check-out |
| `/events/[id]/checkout` | Confirm check-out |
| `/events/[id]/checkin` | Confirm return |
| `/scan` (check-out mode, cart populated) | Check out N items |
| `/scan` (check-in mode, cart populated) | Return N items |
| `/users` | Invite user |
| `/reports/*` | Export CSV |
| `/login` | Sign in |
| `/register` (admin invite) | Create account |
| `/forgot-password` | Send reset link |

### Empty-state copy (Q8)

| Surface | Heading | Body | Action |
|---------|---------|------|--------|
| `/inventory` | No items yet | Add your first inventory item to get started. | `[Add item]` |
| `/events` | No events scheduled | Create an event to begin checking items out. | `[Create event]` |
| `/reports/missing` | Nothing missing | All checked-out items are accounted for. | — |
| `/reports/out` | Nothing checked out | No items are currently at events. | — |
| `/users` | Just you, for now | Invite teammates to check items in and out. | `[Invite user]` |
| `/scan` (no result yet) | Scan an item | Point the camera at a QR code or barcode, or enter a SKU manually. | `[Enter SKU]` |
| Scan cart (empty) | Cart is empty | Scan items to add them to this check-out. | — |

Empty-state visual: centered, vertical stack — 24px lucide icon (muted-foreground) → 18px heading → 14px body (muted-foreground) → action button if present. Vertical padding `py-16` (64px).

### Destructive confirmations (Q9)

All destructive actions use `<AlertDialog/>` (not `<Dialog/>`). Cancel button is `variant="outline"` (left), confirm is `variant="destructive"` (right). The confirm button label is the exact verb of the action, never "OK" or "Yes".

| Action | Dialog title | Dialog body | Confirm label |
|--------|--------------|-------------|---------------|
| Retire item | Retire this item? | It will be removed from active inventory and won't appear in scans or events. Past history is kept. | Retire item |
| Cancel event | Cancel this event? | Items still checked out must be returned manually. The event won't appear in future schedules. | Cancel event |
| Disable user | Disable this user? | They lose access immediately. Their past activity stays in reports. | Disable user |
| Write off item | Write off this item? | It will be marked as a permanent loss and removed from stock counts. This can't be undone. | Write off |

### Error copy (Q9)

| Condition | Heading | Body | Recovery |
|-----------|---------|------|----------|
| Insufficient stock | Not enough stock | Only N available. Reduce the quantity or remove this item. | Inline error on qty field |
| No scan match | Item not recognized | This code isn't in inventory. Check the label or enter the SKU manually. | `[Enter SKU]` button |
| Camera blocked | Camera access needed | Allow camera permission in your browser to scan codes. | `[Enter SKU]` fallback |
| Offline | You're offline | Reconnect to save changes. Your work isn't lost. | Auto-retry on reconnect (Phase 2) |
| Form validation | (inline, per-field) | Specific to field, e.g. "SKU is required." | Inline below field, `text-destructive` |
| Unexpected error | Something went wrong | Refresh the page. If this keeps happening, contact your admin. | `[Refresh]` button |

### Toast copy patterns

- Success: `{Noun} {past-tense verb}` — e.g. "Item added", "Event created", "User invited"
- Error: `Couldn't {verb} {noun}` — e.g. "Couldn't add item", "Couldn't save changes"
- Action toast (with undo): `{Verb}-ed {noun}` + `[Undo]` — e.g. "Item retired" `[Undo]`

## Interaction States (contract)

Every interactive element must define all five states. Implementation must use Tailwind variants, not custom CSS.

| State | Tailwind | Visual |
|-------|----------|--------|
| Default | (base) | Token-driven |
| Hover | `hover:` | `bg-accent` on ghost/outline; brightness shift on primary |
| Focus-visible | `focus-visible:` | 2px `ring` using `--ring` token; `outline-none` on element |
| Active | `active:` | Slight scale or darkened bg, shadcn defaults |
| Disabled | `disabled:` | `opacity-50 pointer-events-none` |

Keyboard access is mandatory. Every action reachable by mouse must be reachable by Tab + Enter. Modal traps focus. Esc closes any overlay.

## Accessibility Floor

- Color contrast minimum WCAG AA: 4.5:1 for body, 3:1 for ≥18px or bold ≥14px. Status dots are decorative; status meaning conveyed by both color and text label.
- All form inputs have associated `<label>` (not placeholder-as-label).
- All icon-only buttons have `aria-label` and a `<Tooltip/>`.
- Touch targets ≥44px on the scanner and qty stepper (see Spacing exceptions).
- Skip-to-content link on the app shell.
- Page `<title>` set per route via Next.js metadata.

## Mock Data Contract (Phase 1 only)

- All data sourced from `lib/mock/<entity>.ts` — typed, exported, deterministic (no `Math.random()` at module scope).
- Mock arrays must contain enough rows to exercise pagination on every list page (≥30 items for inventory, ≥10 for events, ≥5 for users).
- Status distribution in mocks must include at least one item of every status in the palette so empty-state and badge variants are visible during dev.
- No mock fetch delay simulation in Phase 1 — skeletons will be wired in Phase 2 with real Firestore calls.

---

## Out of Scope for Phase 1

These are deferred to Phase 2 or later — do not implement during Phase 1:

- Firebase Auth wiring (login form validates but routes to `/` regardless)
- Firestore reads/writes (all data from `lib/mock/`)
- Server actions
- Real-time listeners
- Email invites for `/users`
- Scanner result persistence (camera decodes, logs to console only)
- Density toggle (locked at "comfortable" 40px)
- Per-user theme persistence (next-themes default localStorage only)
- CSV export logic (button renders but no-ops)
